# Make All Perfect Website — Project Regulation

Этот документ — постоянный регламент проекта Make All Perfect.  
Все новые блоки, страницы, ассеты и правки должны интегрироваться строго по этим правилам.

Главная цель: собирать сайт поэтапно, без хаоса в структуре, без переосмысления дизайна и без потери премиального визуального стиля.

---

## 1. Общий принцип работы

Каждый новый блок или страница добавляется как отдельный готовый пакет:

```text
block-XX-section-name/
  README.md
  integration.md
  code/
  assets/
```

Или, если блок уже адаптирован под Next.js:

```text
block-XX-section-name/
  README.md
  integration.md
  SectionName.tsx
  section-name.module.css
  assets/
```

OpenClove должен не “придумывать дизайн”, а аккуратно интегрировать готовый блок в проект.

---

## 2. Главные запреты

Нельзя:

- переосмысливать дизайн без отдельного указания;
- заменять готовую верстку на шаблон;
- менять композицию, отступы, типографику, цвета, анимации и визуальный стиль самовольно;
- использовать stock-photo;
- использовать дешёвые иконки, clipart, случайные 3D-ассеты;
- вставлять изображения в HTML/React через base64;
- складывать все ассеты в одну папку без структуры;
- называть файлы `image1.png`, `final.png`, `new-final-copy.webp`;
- ломать уже готовые секции при добавлении новых;
- писать глобальные стили, которые могут повлиять на другие блоки.

---

## 3. Визуальный стиль сайта

Сайт Make All Perfect должен сохранять единый стиль:

- Apple-inspired premium minimalism;
- luxury B2B digital agency;
- чистый белый и off-white фон;
- много воздуха;
- крупная графитовая типографика;
- мягкий свет;
- аккуратные тени;
- glassmorphism;
- subtle gradients;
- high-end UI/UX;
- спокойная уверенная подача;
- без визуального шума;
- без ярких кислотных цветов;
- без перегруза декоративными элементами.

Основные цвета:

```text
White: #FFFFFF
Off-white: #F5F6F8 / #F5F5F7
Graphite: #0E0F12 / #1A1C20
Muted gray: #6B6F76
Soft line: rgba(15,16,20,0.08)
Glass white: rgba(255,255,255,0.55–0.78)
Subtle silver/blue accent — только дозированно
```

## 4. Структура проекта для Next.js

Если проект на Next.js, использовать такую структуру:

```text
public/
  assets/
    home/
      hero/
      services/
      cases/
      process/
      cta/
    shared/
      logo/
      icons/

src/
  app/
    page.tsx
    layout.tsx
    globals.css

  components/
    layout/
      Header.tsx
      Footer.tsx

    sections/
      HeroSection.tsx
      ServicesSection.tsx
      CasesSection.tsx
      ProcessSection.tsx
      CtaSection.tsx

    ui/
      Button.tsx

```

Если в проекте уже есть другая структура, можно адаптировать, но логика должна остаться такой же: компоненты отдельно, ассеты отдельно, общие элементы отдельно.

## 6. Именование компонентов

Каждый блок должен быть отдельным компонентом или отдельной секцией:

```text
HeroSection
ServicesSection
CasesSection
ProcessSection
BenefitsSection
CtaSection
Footer
```

Нельзя использовать абстрактные названия вроде:

```text
Section1
Block
NewBlock
FinalSection
```

---

## 7. Именование CSS-классов

Классы должны быть уникальными и понятными.

Хорошо:

```css
.hero-section {}
.hero-inner {}
.hero-title {}
.services-section {}
.services-grid {}
.cases-card {}
```

Плохо:

```css
.section {}
.block {}
.title {}
.card {}
.text {}
```

Причина: общие классы могут сломать другие секции сайта.

---

## 8. Порядок секций на главной странице

Базовый порядок главной страницы:

```text
1. Header
2. HeroSection
3. ServicesSection
4. CasesSection
5. ProcessSection
6. BenefitsSection / WhyUsSection
7. CtaSection
8. Footer
```

Если добавляется новый блок, он вставляется только в указанное место.  
Нельзя самовольно менять порядок секций.

---

## 9. Header

В шапке использовать:

- лёгкий логотип на прозрачном фоне;
- рядом текст: `Make All Perfect`;
- меню;
- CTA-кнопка или бургер справа;
- glassmorphism-эффект;
- аккуратные тени;
- адаптивную мобильную версию.

Текущий выбранный подход для hero/header:

- логотип в шапке — лёгкий PNG/WebP/SVG на прозрачном фоне;
- не использовать тяжёлый 3D-логотип в хедере;
- основной 3D/стеклянный визуал может быть в hero background;
- рядом с логотипом текст `Make All Perfect`.

---

## 10. Hero-блок

Текущий hero-блок считается базовым визуальным ориентиром.

Сохранять:

- Apple-inspired premium minimalism;
- белый/off-white фон;
- крупный H1;
- графитовую типографику;
- две кнопки;
- мягкий glassmorphism;
- премиальный background WebP;
- SVG line icons в glass-кружках;
- чистую композицию без перегруза.

Для нижних иконок hero использовать:

- inline SVG line icons;
- glass-контейнеры;
- не использовать тяжёлые PNG/WebP 3D-иконки для маленьких элементов.

---

## 11. Ассеты

Все изображения должны храниться отдельно, не в base64.

Правильные пути:

```text
public/assets/home/hero/hero-bg.webp
public/assets/home/hero/logo-header.webp
public/assets/home/services/service-01.webp
public/assets/shared/logo/logo-header.webp
public/assets/shared/icons/icon-strategy.svg
```

Для обычного HTML:

```text
assets/home/hero/hero-bg.webp
assets/shared/logo/logo-header.webp
```

---

## 12. Именование ассетов

Использовать kebab-case:

```text
hero-bg.webp
logo-header.webp
icon-strategy.svg
icon-design.svg
icon-analytics.svg
service-performance.webp
case-marketplace-growth.webp
```

Не использовать:

```text
image1.png
photo.png
final-final.png
logo_new_copy_2.webp
```

---

## 13. Оптимизация изображений

Общие правила:

- не вставлять base64;
- тяжёлые изображения конвертировать в WebP;
- для прозрачных простых элементов использовать SVG, если возможно;
- для маленьких иконок не использовать большие 3D PNG;
- не пережимать изображения до дешёвого вида.

Целевые ориентиры:

```text
Hero background: 50–150 KB
Header logo: 5–40 KB
Small decorative image: 5–50 KB
SVG icon: 1–5 KB
Full first screen target: ideally under 500 KB–1 MB
```

Для hero background важен баланс: лучше немного больше вес, но премиальный вид без грязных градиентов.

---

## 14. SVG-иконки

Для маленьких интерфейсных элементов предпочтительны SVG line icons.

Стиль:

```css
stroke: #4B5563;
stroke-width: 1.8;
fill: none;
stroke-linecap: round;
stroke-linejoin: round;
```

Иконки можно размещать inline внутри HTML/React, если это не засоряет код.

---

## 15. CSS и стили

Если проект на Next.js:

- предпочтительно использовать CSS Modules для секций;
- глобальные стили только для базовых переменных, reset, body, typography;
- не добавлять стили, которые могут случайно изменить другие блоки.

Если проект на HTML:

- у каждой секции должен быть отдельный CSS-файл или отдельный явно подписанный блок стилей;
- классы должны быть уникальными.

---

## 16. Адаптив

Каждый блок обязательно проверять на:

```text
Desktop: 1440px+
Laptop: 1280px
Tablet: 768px
Mobile: 375–430px
```

Проверить:

- нет горизонтального скролла;
- текст не вылезает за экран;
- кнопки не ломаются;
- изображения не перекрывают важный текст;
- header не перекрывает hero-контент;
- меню работает на мобильном.

---

## 17. Анимации

Анимации должны быть:

- мягкие;
- спокойные;
- премиальные;
- без резких прыжков;
- без дешёвых эффектов.

Разрешено:

- fade-up;
- subtle hover;
- smooth transform;
- мягкий scale/translate;
- light sheen на кнопках.

Нельзя:

- агрессивные bounce;
- glitch;
- кислотные neon effects;
- резкие zoom;
- перегруженные parallax-эффекты.

Всегда учитывать `prefers-reduced-motion`.

---

## 18. Кнопки

Кнопки должны быть чистыми и премиальными.

Основная кнопка:

- графитовая/чёрная;
- белый текст;
- мягкая тень;
- аккуратный hover.

Вторичная кнопка:

- белая/glass;
- тонкая граница;
- графитовый текст.

---

## 19. Интеграция нового блока

Каждый новый блок должен идти с файлом `integration.md`.

В `integration.md` обязательно указать:

```text
1. Название блока
2. Куда вставить
3. Какие файлы добавить
4. Какие файлы изменить
5. Куда положить ассеты
6. Как импортировать компонент
7. Что проверить после интеграции
8. Какие визуальные элементы нельзя менять
```

---

## 20. Формат задачи для OpenClove

При интеграции нового блока использовать такой формат задачи:

```text
Интегрируй новый блок строго по PROJECT_REGULATION.md и integration.md.

Не переосмысливай дизайн.
Не меняй композицию, стили, отступы, цвета и анимации без необходимости.
Не используй base64.
Все ассеты положи в правильные папки.
Проверь desktop/tablet/mobile.
Проверь Vercel build.
После работы пришли список изменённых файлов, ссылку на commit и Vercel preview.
```

---

## 21. Отчёт после интеграции

После каждой интеграции OpenClove должен прислать:

```text
1. Что изменено
2. Какие файлы добавлены
3. Какие файлы изменены
4. Какие ассеты добавлены
5. Ссылку на GitHub commit
6. Ссылку на Vercel preview
7. Что проверить вручную
8. Если что-то не перенесено 1 в 1 — почему
```

---

## 22. Контроль качества

Перед завершением каждой задачи проверить:

- проект собирается без ошибок;
- Vercel build проходит;
- нет битых путей к изображениям;
- нет горизонтального скролла;
- нет console errors;
- секция визуально совпадает с исходником;
- mobile не сломан;
- стили не затронули другие блоки;
- ассеты оптимизированы;
- нет base64 в коде.

---

## 23. SEO и доступность

Базовые правила:

- один H1 на странице;
- у изображений, если они значимые, должен быть alt;
- декоративные изображения могут иметь `alt=""`;
- кнопки и ссылки должны быть понятными;
- меню должно иметь aria-label;
- burger должен иметь aria-expanded;
- структура заголовков должна быть логичной.

---

## 24. Что делать, если есть сомнение

Если OpenClove не уверен, куда вставлять блок или как назвать файл:

- не придумывать хаотично;
- использовать существующую структуру проекта;
- если структуры нет — создать по этому регламенту;
- если есть риск сломать готовую секцию — остановиться и сообщить.

---

## 25. Главная установка

Make All Perfect — это сайт маркетингового агентства полного цикла.  
Он должен выглядеть дорого, спокойно, уверенно и современно.

Главный критерий: не “просто красиво”, а **премиальный B2B-сайт, который вызывает доверие у владельцев бизнеса и маркетинг-директоров**.
